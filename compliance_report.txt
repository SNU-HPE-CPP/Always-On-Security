    - node3 — queue mgr (10.10.1.23)
    - Network: compute-net (10.10.1.0/24)
    - User: non-root UID 10001
    - No Docker socket / No HMAC key

  [Data Storage Zone (§2.1.2)]
    - node4 — data/storage (10.10.2.31)
    - Network: storage-net (10.10.2.0/24)
    - Isolated from compute-net
    - Image digest attested
    - Config integrity monitored

  [Management Zone (§2.1.7)]
    - controller — security bus (10.10.3.10)
    - risk-engine — scoring (10.10.3.11)
    - dashboard — UI (10.10.3.20)
    - alert_ingestor — SIEM (10.10.3.40)
    - Network: mgmt-net (10.10.3.0/24, internal)

  [Infrastructure Monitoring Zone]
    - host-observer — image/drift (10.10.3.12)
    - security-monitor — NIDS/NADS (10.10.1/2/3.250)
    - docker-socket-proxy — API gateway
    - HMAC-signed telemetry to controller
    - Out-of-band (no exec into nodes)

  [Access Zone (§2.1.6)]
    - dashboard — web UI (port 5000)
    - Only service with external port binding
    - Read-only DB access (shared_data:ro)
    - Gap: no MFA / auth proxy

  [Network Fabric]
    - 3 isolated bridge networks
    - All compute/storage/mgmt: internal: true
    - host-access bridge (dashboard only)
    - Docker socket: proxy-mediated only
    - No direct internet access for workloads

7. IDENTIFIED GAPS & RECOMMENDATIONS
================================================================================

  GAP-01: Access Zone Authentication  (Priority: HIGH)
  NIST Reference: §3.2.1 Access Zone Threats, §4.1
--------------------------------------------------------------------------------
    The security dashboard (port 5000) is accessible without authentication.
    NIST SP 800-223 §3.2.1 specifically flags that applying multi-factor
    authentication (MFA) to HPC system access is a proven method to mitigate the
    risk of unauthorised access. In production this represents a direct access-
    zone exposure.

  Recommendation:
    Deploy an OAuth2 reverse proxy (OAuth2-proxy, Authentik, or Keycloak) in
    front of the dashboard, or add session-based authentication directly to the
    Flask application.

  GAP-02: Compute Node Sanitization Between Jobs  (Priority: MEDIUM)
  NIST Reference: §4.2 Compute Node Sanitization
--------------------------------------------------------------------------------
    NIST §4.2 recommends node health checks, forced reboots for unhealthy nodes,
    GPU scrubbing, memory wiping, and firmware validation between tenant job
    runs. The current system detects runtime drift but does not perform active
    sanitization: no trigger to wipe GPU state, clear memory, or reboot between
    jobs.

  Recommendation:
    Implement a job epilogue hook in the node_agent that triggers container
    reconstruction (stop + rm + recreate from approved image digest) after each
    job, and integrate with host-observer to confirm sanitization before the
    next job.

  GAP-03: Container Image CVE Scanning  (Priority: MEDIUM)
  NIST Reference: §4.4 Securing Containers, §3.3 Supply Chain Threats
--------------------------------------------------------------------------------
    Approved image digests are attested against a static baseline
    (approved_images.yaml) but images are not scanned for known CVEs at
    build/pull time. NIST §4.4 notes that container contents may not be
    observable by some security auditing tools and recommends tools like Qualys
    or Anchore for supply-chain auditing.

  Recommendation:
    Integrate Trivy or Grype into the CI pipeline to scan node images at build
    time. Block image approval if high-severity CVEs are present. Re-scan
    periodically against updated vulnerability databases.


  GAP-04: Falco Kernel Probe Compatibility — ** RESOLVED **
  NIST Reference: §3.2.3 HPC Computing Zone Threats
--------------------------------------------------------------------------------
    Falco 0.44.1 with the modern eBPF driver is confirmed operational on Linux
    6.11.0-19-generic (Ubuntu 24.04). Falco runs as a host-native service
    writing structured JSON to /var/log/falco/events.json. The security-monitor
    container mounts this path read-only and falco_collector.py tails the event
    stream in real time, providing full kernel-level syscall detection
    (CONTAINER_ESCAPE_ATTEMPT, REVERSE_SHELL, PRIV_ESC_ATTEMPT) without
    requiring a containerised Falco process.

  Evidence: Live events confirmed in /var/log/falco/events.json and processed by
  falco_collector.py (security-monitor logs confirm tailing at startup).
